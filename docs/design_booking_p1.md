## 1. 背景と診断（2026-07-14 実サイト調査で確定した事実）

- 現行 yah.homes の予約ページ（/booking/kiyokawa/ 等）は **Beds24のホスト型予約画面を iframe 埋め込み**している:
  `<iframe src="https://beds24.com/booking2.php?propid=278158&referer=iframe">`
- これが「遅い・スマホで崩れる・デザインが浮く」の直接原因。**API連携は現状存在しない**（画面の間借りのみ）。
- **iframe内で完結する予約はGA4/Google Adsから計測不能**。現在Google広告のCVは「Airbnbリンククリック」を代用している（2026-07-14時点、日本向けキャンペーン稼働中: CTR 12.8%・CPC約¥91）。
- サイトは Astro 製・多言語パスあり（/ja /ko /zh /th ＋ 既定en）・GA4 導入済み（G-VJ5DDRML79）。
- 物件: 清川（最大7名・寝室3・浴室1階・那珂川沿い・専用駐車場・★4.77/47件）・高砂（最大6名・寝室3・洗面台3・渡辺通駅徒歩5-10分・大型車可・★4.67/36件）。**今後5棟へ拡大予定**。
- 予約チャネル: Airbnb（メイン）＋ Booking.com。チャネル同期は Beds24。

## 2. 決定事項（アーキテクチャ原則）

1. **在庫・料金の唯一の真実 = Beds24**。チャネルマネージャ（OTA同期・ダブルブッキング防止）は借り続ける。フルスクラッチはしない（30棟未満では投資対効果が合わない）。
2. **予約UI・決済・顧客データ・計測 = 自社**。iframeを撤去し、自社画面から Beds24 API v2 を叩く。
3. **運営会社の業務は一切変えない**。自社サイト経由の予約は Beds24 上で他OTAの予約と同じ見え方になる。
4. **顧客データは Firestore に全チャネル分ミラー**（Beds24 Webhook 経由）。これはUI改修と独立して即着手可能。
5. **段階導入（§4）**。各段は次の段の土台になり、捨てる作業がない。ロールバックは常に「前の段に戻す」だけ。

## 2.5 事業KPI（このプロジェクトが何のカネを取りに行くか）

- **直販比率**: 現状ほぼ0〜10%（広告経由もAirbnbリンク送客のため）→ **12ヶ月で35% → 18ヶ月で50%**。
  OTA実効手数料 11.25%（75%×15%）→ 7.5% ＝ **5棟85%巡航で年+287万円**（初年度モデルで+191万）。
- 直販比率の積み上げ計画: ①広告の着地を自社予約へ（段1'・寄与+17〜25%）②リピーター直販クーポン（D1メール・+8〜12%）③指名検索の刈り取り（+3〜5%）④室内QRカード（+2〜3%）。
- **直販ベストレート原則**: 直販が常に最安 or 特典付き（OTA比5%安／レイトチェックアウト等）。
  Booking.comのレートパリティは「割引コード・会員価格・特典」形式で実務対応。
- **OTAは敵にしない**: OTA＝新規獲得の広告費15%と再定義。新規はOTAで獲り、2回目から直販へ。
  Airbnbランキング（新規の蛇口）を痛める撤退戦はしない。
- 運営体制の決定事項（2026-07-14）: **現管理会社は維持**（12%は相場15〜20%比で良心的・関係良好）。
  5棟化の契約時に「定型連絡の自社自動化（§9）＋5棟全委託」と引き換えに**10%へのスコープ交渉**を行う。
  分解外注（メッセージ代行＋セコム＋清掃直契約）は実行案ではなく**BATNA（交渉の物差し・有事のバックアップ）**として見積もりのみ取得。

## 3. ページ構造（IA）—「売るページは棟ごと・予約は1枚」（ホテル型）

```
 Home（/）            玄関: ブランド・棟の顔見せ・日付×人数検索ボックス
   ├ /properties/kiyokawa/  ② 物件詳細 =「売る」ページ（棟ごと・軽さ最優先）
   ├ /properties/takasago/     写真・設備・レビュー・地図・周辺 ＋ ミニ予約ウィジェットのみ
   │      │ CTA（棟・日付を引き継いで遷移）
   │      ▼
   └ /book/           ③ 予約ページ =「確定する」ページ（全棟共通・1枚だけ）
                         日付×人数 → 泊まれる棟をホテルの部屋選択のように一覧
                         （清川 ¥XX,XXX総額 / 高砂 ¥XX,XXX総額 / 満室棟は明示）
                         → 棟を選んで確定へ（段0暫定: Beds24ページ／段1': 自社決済）
```

- **予約フローは全棟で /book/ の1枚に統合**（ホテルの「部屋を選ぶ」モデル）。棟が5に増えても /book/ は無変更で、②のページを足すだけ。
- **クロスセルが構造で効く**: 目当ての棟が満室でも、その場で他棟の空きが見える（棟別予約ページでは離脱していた客を拾う）。
- ②からの遷移は `/book/?prop={slug}&checkin=…&checkout=…&guests=…` で**棟選択済み・日付入力済み**の状態で開く（選び直しの手間ゼロ）。
- **②は1ページ1仕事**: 売ることに専念。フル予約フォームは置かない。SEO/広告の着地は必ず②。
- 既存 `/booking/{slug}/` は `/properties/{slug}/` へ**301リダイレクト**（URL意味論の修正。「book」は③に譲る）。
- ※旧案にあった「① 横断検索ページ」は /book/ に吸収（検索＝予約入口として1枚で兼ねる）。
- ②のミニ予約ウィジェット仕様:
  - PC: 右カラム sticky カード（料金〜/泊・チェックイン/アウト・人数・総額・CTA）
  - スマホ: 画面下部固定バー（「¥XX,XXX〜 [空室を見る]」）
  - カレンダーJSは**ユーザーが日付欄に触れてから遅延ロード**（②の初期表示速度を犠牲にしない）

## 3.5 ストアフロントUXパターン集（大手OTAから真似るもの・避けるもの）

②物件詳細・③/book/ の実装時に適用する。優先度順（★=最優先・実装が軽くCVR直結）。

### 採用するパターン

1. ★**総額表示**: 「¥45,000/泊」でなく「2泊6名 合計¥98,000（清掃料込み）」を最初から表示。内訳（泊数×単価＋清掃料＋税）は畳んで開示。手数料の後出しは絶対にしない。**直販の総額先出しはOTAへの差別化そのもの**。
2. ★**キャンセルポリシーの1行明示**: 料金直下に「◯月◯日まで無料キャンセル」を日付入り1行で。全文はリンク先。家族客の最後のためらい対策。
3. **寝室ごとのベッド構成カード**（Airbnb: Sleeping arrangements）: 「寝室1: ダブル1／寝室2: シングル2…」を部屋カードで。6〜7名グループ・三世代客は「誰がどこで寝るか」で決める。部屋割りの提案例（3世代なら…）まで書く。
4. **カテゴリ別レビュー＋Airbnbレビュー転載**: 総合★だけでなく内訳（清潔さ/立地/チェックイン）バー＋レビュー本文（国・時期つき）を各棟10件以上。直販の「ここ大丈夫？」をAirbnbで積んだ資産（★4.77/47・★4.67/36）で潰す。
5. **部屋ラベル付き写真ギャラリー**（Vrbo: 部屋別ツアー）: 「リビング/寝室1/浴室/駐車場」でセクション分け。洗面台3（高砂）等の武器は写真で探せる状態に。
6. **設備の構造化リスト＋家族向けバッジ**: ベビーベッド・ベッドガード・子ども椅子・洗濯乾燥機を独立セクションに（全市場で家族系クエリがYoY∞成長の実査結果に対応）。同内容を VacationRental schema に焼き込み GEO にも効かせる。
7. **アカウント不要のゲスト予約**（段1'）: Airbnbは登録必須＝大手が構造的にできない差別化。メールだけで予約完了させる。
8. 余裕があれば: 地図＋「ここから何分」（柳橋7分等のコーパス流用）／総額の通貨並記（≈$・₩・NT$）／Self check-in バッジ／「直販が一番お得」比較表（OTA vs 直販: 価格・特典・柔軟性）。

### 避けるパターン（大手の悪癖）

- ❌ 偽の緊急性（「今20人が見ています」・常時「残り1室」）。**本当に残り1棟の時だけ**表示するのは可。虚偽表示はレビュー資産を毀損する。
- ❌ 会員登録の強制・ログイン誘導。
- ❌ クロスセル物量（保険・レンタカー等）。1ページ1仕事に反する。
- ❌ 手数料の後出し（採用パターン1の逆）。

## 5. 段0 仕様

### 5-1. iframe撤去
- `/booking/{slug}/` の iframe セクションを削除し、CTAボタンに置換:
  `https://beds24.com/booking2.php?propid={propid}`（**新規タブで開く**）
- 文言例: 「空室カレンダー・ご予約はこちら（安全な予約ページが開きます）」
- Airbnbリンクは併置のまま（当面の信頼担保・比較手段）。

### 5-2. Webhook → Firestore ミラー
- Beds24 管理画面で Webhook を設定（予約の作成/変更/キャンセル時に通知）。
- 受け口: `functions` に HTTPS 関数 `beds24Webhook` を追加。
  - 検証: 共有シークレット（クエリ or ヘッダ）で照合。不一致は 403。
  - 受信後、Beds24 API `GET /bookings/{id}` で正データを取り直してから保存（Webhookペイロードを直接信用しない）。
- Firestore スキーマ:

```
bookings/{beds24BookingId}
  propKey        "kiyokawa" | "takasago" | …
  channel        "direct" | "airbnb" | "booking" | …
  status         confirmed | cancelled | modified …（Beds24のstatusを写像）
  checkIn/checkOut  ISO date
  numAdult/numChild
  amount         合計金額（数値・JPY）
  guest: { name, email, phone, country, lang }
  createdAt/updatedAt  epoch ms
  raw            Beds24レスポンス原文（スキーマ変化に備え保全）

guests/{emailLower}          ← 顧客台帳（リピート判定の主キーはemail小文字）
  name, country, lang
  bookingIds: [ ... ]
  firstStay / lastStay / totalStays / totalAmount

events/{auto}                ← 既存の計測系と整合させる
```

- firestore.rules: `bookings` `guests` は**クライアント読み書き禁止**（Functionsのみ・admin経由）。個人情報のため公開読み取りは絶対に付けない。

## 7. 段1' 仕様 — /book/ フル実装（旧段2+3を統合）

### 7-1. 読み取りAPI（Functions プロキシ）
- Functions に `bookingApi`（HTTPS）を追加。クライアントは自社ドメインの同関数のみ叩く（**BEDS24トークンをクライアントに出さない**）。
- エンドポイント（自社側）:
  - `GET /api/availability?prop={slug}&from=&to=` → 日別の空/塞・最低泊数
  - `GET /api/quote?prop={slug}&checkin=&checkout=&adults=&children=` → 総額・内訳
- Beds24 API v2 を利用（token認証）。代表エンドポイントは在庫カレンダー系・料金オファー系・予約系（**正確なパス/パラメータは実装時に公式 API v2 ドキュメントで最終確認**。v1とv2で体系が違うため注意）。
- キャッシュ: 空室・料金は **5分キャッシュ**（Functionsメモリ＋`Cache-Control`）。予約直前の正はBeds24側で担保されるため表示用途は5分で実用十分。
- レート制限対策: 呼び出しはFunctions集約＋キャッシュで吸収。429時は指数バックオフ。

### 7-2. /book/（自社・全棟共通の予約ページ）— 予約完結までのフルフロー
1. 日付×人数フォーム（②から来た場合はURLパラメータで入力済み・棟も選択済み表示）
2. `/api/availability`＋`/api/quote` を**全棟並列**で叩き、泊まれる棟をカード一覧（総額・定員・満室明示・§3.5の総額表示原則）
3. 棟を選ぶ → 予約内容確認 → ゲスト情報入力（宿泊者名簿要件込み・**アカウント登録不要**）→ Stripe決済
4. 決済フロー: Stripe PaymentIntent（オーソリ）→ 成功後 `POST /bookings`（Beds24へ書き込み）→ **書込成功で初めてキャプチャ・確定** → 確認メール＋Firestoreミラー
5. 書込失敗（OTAとの同時性衝突・レアケース）: 自動でオーソリ解放（返金）＋代替日提案画面。**このフォールバックは必須実装**
- 段0の暫定期間のみ: ②のCTAはBeds24ホストページへ直リンク。**清川・高砂が同一propid内の「部屋」なら、暫定でも「日付→棟選択」のホテル型1ページUXになる**（別propidならリンク2本で妥協・数週間の暫定なので投資しない → §14-2）。

## 9. ゲスト連絡（運営会社の業務設計）

- 直予約ゲストはAirbnbメッセージに存在しない → **Beds24 の自動アクション（Auto Actions）でメール自動送信**に統一。新ツール導入なし。
- テンプレ4通 × 5言語（ja/en/ko/zh-TW/th）:
  1. 予約確認（即時）: 内容・総額・キャンセルポリシー
  2. チェックイン案内（3日前）: 住所・鍵・手順・駐車場（清川）/駅からの道順（高砂）
  3. 当日リマインド: 緊急連絡先
  4. チェックアウト後（翌日）: お礼＋レビュー依頼＋**次回直予約クーポン**
- **要確認**: 現在の連絡運用がどのパターンか（A: Airbnbアプリ手動 / B: Beds24自動アクション設定済み / C: Beds24統合受信箱）。Bなら適用チャネルに直予約を足すだけ。

### 9-2. 対応の3層モデルとエスカレーション（クレーム・緊急時）

怒っているゲスト・深夜緊急は3層で処理する（例: 深夜「お湯が出ない」）:

| 層 | 仕事 | 担当 |
|---|---|---|
| ①コミュニケーション | 一次対応・謝罪・聞き取り・経過連絡・翌朝フォロー | 管理会社（現体制）／将来はメッセージ代行も可 |
| ②物理対応 | 現地駆けつけ・解錠・設備の一次確認・業者手配 | 管理会社（現体制） |
| ③経営判断 | 返金・補償額・退去・警察 | オーナー。ただし下のルール化で月1件程度まで削減 |

**エスカレーション表（事前権限移譲・運用開始時に管理会社と合意）**:

```
・設備不良で入浴不可 → 銭湯代 ¥2,000×人数まで現場判断で即補償OK
・近隣からの騒音クレーム → 1回目は現場が注意連絡 → 改善なければオーナーへ
・¥10,000 までの補償・部分返金 → 現場判断OK（月次報告）
・全額返金・退去・警察・けが人 → 即時オーナーへ電話
```

軽微な判断を現場に渡すことで、ゲストは「その場で解決」を体験し（レビュー防衛）、
オーナーに届くのは本物の経営判断のみになる。
