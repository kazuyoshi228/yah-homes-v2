SEO TECHNICAL AUDIT & MIGRATION PLAN

yah.homes を Google 検索で見つかるサイトにする
— 現状診断と Firebase 移行での改善点

google / organic 月間28ユーザーからの脱却に向けた、実測に基づく技術レポート
調査日 2026年7月11日 対象 https://yah.homes/（Manusホスティング） 調査方法 本番サイト実測・ソースコード精読・GA4/Search Console
要約 — 3つの結論
  1. 現サイトは検索エンジンに「中身が空のページ」として見えています。全URLが同一の空HTMLを返すクライアントレンダリング構成で、本文・見出し・画像がHTMLに一切存在しません。
  2. ボット向けメタ注入にバグがあり、Googlebotには全ページが同じ英語タイトルで返っています。韓国語・中国語ページの検索結果が機能しない直接の原因で、修正自体は1行です（応急処置は現環境でも可能）。
  3. Firebase移行の価値は「全ページを本物の静的HTMLとして配信できること」です。ただし技術改善は土台であり、organic流入の本丸は magazine.yah.mobi をコンテンツエンジンにした検索需要の獲得です。

02技術診断 — 検索エンジンから見た yah.homes

現サイトは Manus のテンプレート（React 19 + Vite + Express + tRPC + Drizzle）で、cname.manus.space 上で稼働しています。実測とソース精読で確認した問題を深刻度順に示します。

深刻度問題実測値・根拠
致命的 本文がHTMLに存在しない（完全CSR） 全URLが同一391KBのシェルを返し、body は <div id="root"></div> のみ。h1・h2・img・本文テキストがゼロ。内容表示に795KBのJS実行が必須
致命的 ボット向けタイトルが全ページ英語ホーム用（バグ） Googlebot UAで /ko/・/about・/booking/kiyokawa を取得すると、すべて同一の英語ホームタイトルが返る（下記コールアウト参照）
高 応答が遅く、HTMLが一切キャッシュされない TTFB 1.2〜2.4秒（目安は0.8秒以下）。HTMLは cache-control: no-cache, no-store で Cloudflare も素通し（cf-cache-status: DYNAMIC）
高 UA判定でボットにだけ別HTMLを返す設計 prerender.ts が既知ボットのUAにのみメタを注入。ユーザーとクローラーで内容が異なる「クローキング」に近い構成で、脆く、検知漏れボット（新興AI検索等）には空シェルが渡る
中 sitemap と実体の不整合 sitemap は /zh-cn/（簡体字）を宣言するが、ルートも翻訳も未実装でSPAフォールバックの200が返る。canonical も通常UAでは /ko、ボットには /ko/ と不統一
中 検索面が構造的に狭い 実質7ページ × 4言語のブランドサイトのみ。「福岡 一棟貸し」「후쿠오카 숙소」等の検索需要を受けるコンテンツページが存在しない
確認済みバグ — 全ページが英語ホームのタイトルに

server/_core/prerender.ts はパスごとのタイトル・説明文を丁寧に定義していますが（韓国語・繁体字・タイ語まで完備）、ミドルウェアを app.use("*", ...) に載せているため req.path が常に "/" になり、どのURLでもホーム用メタがマッチします。結果、Googlebot には全ページが「yah.homes | Whole-House Villa Rental Fukuoka — …」で返っており、これは実測で確認済みです。全ページ重複タイトル＝Googleからは「同じページの複製群」に見え、韓国語クエリでの表示は絶望的です。修正は req.path → req.originalUrl（クエリ除去）の実質1行で、移行を待たず今日直せます。

維持すべき資産 — ここは既によくできている

robots.txt（AIクローラー明示許可）、hreflang完備のsitemap、リッチなJSON-LD（LodgingBusiness・FAQPage・AggregateRating・Review）、llms.txt / llms-full.txt。これらの設計思想はそのまま新環境へ移植する価値があります。問題は「何を書くか」ではなく「どう配信するか」の層にあります。

03Search Console 実データ — インデックスの詰まり

Search Console（sc-domain:yah.homes）を実際に確認したところ、CSRの問題がインデックス登録に具体的な形で表れていました。

指標値
既知ページ数25
インデックス登録済み12
未登録（うち「ページにリダイレクトがあります」）13（11）
直近3ヶ月の検索クエリ数14件
同期間のクリック数0
確認済みの構造的問題 — JSリダイレクトが未インデックスを量産

/home・/ko/home・/zh/home などに実際にアクセスすると、HTTPステータスは200番台（リダイレクトではない）ですが、ページ内のReactルーター（wouterの<Redirect>）がブラウザ内で / へ転送します。サーバーが「これはリダイレクト」と最初から宣言しないため、GooglebotはJSを実行して初めてリダイレクトだと判定でき、その間ページは「クロール済み・リダイレクトあり・未インデックス」で足止めされます。これがSearch Console上の未登録13件中11件の正体です。

さらに悪いことに、レガシーパス /korean は実サーバー側で301により /ko/home へ飛びますが、その /ko/home 自体が上記のJSリダイレクトで /ko/ へ二段階目の転送をしています。1回のアクセスに2ホップかかり、クロール予算を余計に消費する構成です（www.yah.homes → yah.homes、http:// → https:// はサーバー側の正しい301で、ここは問題ありません）。

検索パフォーマンスも同じ実態を裏づけています。過去3ヶ月で表示されたクエリはわずか14件、「福岡包棟民宿推薦」（中国語検索）のような明確な需要語も表示回数が数件どまりで、クリックは全期間ゼロ。表示されているクエリの多くもブランド名の揺れ（villa yahachi garden 等）で、指名検索すら弱い状態です。CSRによってGoogleがコンテンツをほぼ評価できていない、という02章の診断と整合します。

04Firebase 移行で何が変わるか

magazine.yah.mobi で確立した BaaS-first 構成（Firestore / Auth / Hosting / Functions）に寄せることで、上記の致命的問題はアーキテクチャごと消滅します。核心はビルド時に全ページ（ページ × 言語のマトリクス）を完全な静的HTMLとして生成し、Firebase Hosting のCDNから配信することです。

現状（Manus / Express + CSR）

  • 全URLで同一の空シェルHTML（391KB）
  • 本文表示にJS 795KBの実行が必須
  • ボットUA判定でメタだけ差し替え（バグあり）
  • TTFB 1.2〜2.4秒・HTMLキャッシュなし
  • Expressサーバー常駐・Manusに依存

移行後（Firebase Hosting + SSG）

  • URLごとに本文・h1・画像入りの完全なHTML
  • title / meta / hreflang / canonical / JSON-LD をビルド時に焼き込み — 全UAに同一配信
  • CDNエッジ配信で TTFB 100ms前後・CWV改善
  • 問い合わせ・Beds24連携は Functions へ移植
  • magazine と同一の運用基盤・デプロイフロー

改善マッピング

診断した問題Firebase構成での解決
JSリダイレクトで11ページが未インデックスSSGでは /home 系レガシーパスをビルド設定またはHostingの redirects 設定で実HTTP 301に統一。JS実行に依存しないため1ホップで完結し、Googlebotが即座に判定できる
本文がHTMLにないSSG（Astro または Next.js の静的出力）で全28ページ（7ページ×4言語）を事前レンダリング。クローラーもAI検索もJS実行なしで全文を読める
ボット向け注入バグ・クローキングUA判定という仕組み自体を廃止。人間とボットが同じHTMLを見るため、バグの入る余地がない
遅いTTFB・キャッシュなしHosting のCDN配信＋アセットは immutable キャッシュ。HTMLも再デプロイまでエッジキャッシュ可能
sitemap不整合sitemap・hreflang・canonical をビルドで自動生成し、実在ページと機械的に一致させる。/zh-cn/ は実装するか宣言から外すかをこの時点で確定
検索面が狭いmagazine.yah.mobi（Firestore CMS）をコンテンツエンジンとして連携 — 次章
正直な注意点

技術移行だけでは organic は劇的には増えません。7ページのブランドサイトが受けられる検索クエリは指名検索と少数の地域クエリに限られます。移行の意味は「Googleに正しく評価される土台を作ること」と「コンテンツを積む場所を用意すること」の2つです。数字を動かす主役はコンテンツ（P2）です。

05実行ロードマップ

P0

応急処置 — 現環境のまま、移行を待たない

今日〜3日 / 現Manus環境で完結
  • prerender.ts のパスマッチを修正（req.path → req.originalUrl）し、全ページに正しい言語別タイトルを返す
  • /home /ko/home /zh/home /th/home /korean をExpress側で実HTTP 301（現状のJSリダイレクトではなく）に変更。2ホップを1ホップに短縮し、Search Consoleで11件詰まっている未インデックスを解消
  • sitemap から /zh-cn/ を除去（実装するまで宣言しない）、canonical の末尾スラッシュを sitemap と統一
  • Search Console で修正後に該当URLのインデックス再登録をリクエスト
P1

Firebase 移行 — 静的HTML化と配信基盤

1〜2週間
  • Astro（推奨：コンテンツサイト向きで出力が最軽量）または Next.js 静的出力で7ページ×4言語を再構築。既存のReactコンポーネント・翻訳ファイル・JSON-LDは大部分を流用可能
  • Firebase Hosting へデプロイ。画像はAVIF/WebP＋幅別バリアントでLCP 2.5秒以内を目標
  • 問い合わせフォーム・Beds24 webhook・管理画面APIを Cloud Functions へ移植（magazine で実証済みのパターン）
  • DNSを manus.space から Firebase へ切替。移行前後でURL構造を変えないため、リダイレクト設計は最小で済む
P2

コンテンツエンジン — organicの本丸

移行後、継続的に
  • magazine.yah.mobi を検索需要の受け皿に：韓国語・英語での「福岡 家族旅行」「후쿠오카 단독주택 숙소」「fukuoka family accommodation」系ガイド記事を月数本のペースで積む
  • 記事 → yah.homes 予約ページへの内部リンクを設計（既存の /locals（清川ローカルガイド16選）は良い雛形。言語別に拡充する価値が高い）
  • PR TIMES・twitfukuoka 等、既に referral 実績のある媒体からの被リンクを継続獲得
P3

計測の衛生 — 効果を正しく測れる状態に

P1と並行
  • 広告・SNSリンクに必ずUTMを付与（前回のボット問題で「広告と自然流入を区別できない」ことが実害になったのは記憶に新しいところです）
  • GA4のボット汚染対策（Cloudflare のボット対策ルール、またはDirect除外セグメントの常用）
  • Search Console をKPIの一次ソースに：表示回数→クリック→掲載順位の順で改善を追う
