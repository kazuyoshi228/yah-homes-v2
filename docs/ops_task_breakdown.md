# 運営業務 分解表 — yah.homes（5棟・稼働85%・月64組想定）

> 作成: 2026-07-14。関連文書: [design_booking_p1.md](design_booking_p1.md)（予約基盤）・[bcp_operator_risk.md](bcp_operator_risk.md)（運営会社リスク管理）
>
> **用途（3通り）**
> 1. 運営会社との料率交渉（12→10%）: B3/B4/D1 が 運営会社→自動化 に移る「業務分担の新旧比較」として
> 2. パイロット運営会社の要件書: B〜E の 運営会社 列がそのまま委託範囲定義
> 3. BCP: 「自社も持っておくべき知識」（C6業者リスト・E4写真報告・ハウスマニュアル）の洗い出し
>
> 頻度は5棟・85%稼働（月64組・127.5泊）の推計。2棟なら約1/2.5。

```
・設備不良で入浴不可 → 銭湯代 ¥2,000×人数まで現場判断で即補償OK

```

### 構成Y: AIチャット＋本人電話型（2026-07-14決定の最終形）

問い合わせの75〜85%（予約前質問・使い方・変更手続き）をAIチャットが処理し、
電話・最終判断は山田和義本人が担う（実質 月5〜10件・深夜0〜1件）。

| 費目 | 月額目安 | 備考 |
|---|---:|---|
| AIチャット（自社構築・API費） | ¥1万未満 | ハウスガイド5言語＋FAQ＋Beds24空室API(MS2)を知識ベースに。会話ログはコーパス資産（v9のコンシェルジュログ） |
| 電話代行（不在時転送の保険） | ¥1〜3万 | 常用しない。旅行・病気時にワンタップ切替 |
| セコム等（物理駆けつけ・AI/音声から直接発火） | ¥5〜8万 | |
| 清掃直契約（¥11,000×64） | ¥70.4万 | |
| 記帳代行 | ¥3〜5万 | |
| **計** | **約¥80〜90万** | 委託版比 **月¥80〜90万の圧縮** |

**構成Yの成立条件（3つ・外すと24時間拘束になる）**
1. 電話は「最後の窓口」: 導線はチャット→メール→電話。電話には自動音声の振り分け（鍵=セコム案内／変更=チャット誘導／緊急=本人）
2. 不在モード: 電話代行への転送を最初から用意（1人オンコールの弱点対策）
3. 見直し基準: 本人への電話が週3本超×2ヶ月続いたら設計を見直す（AI強化 or 代行範囲拡大）

**注意**: AirbnbゲストのメッセージはAirbnbプラットフォーム内が原則のため、AIチャットの主対象は
直販ゲスト＋予約前のサイト訪問者。直販比率50%計画（design_booking_p1.md §2.5）と同方向。
AIの誤答リスクは「補償・返金・緊急は即エスカレ」のH項エスカレ表で封じる。

## 開業後の運用ノート

- 開業直後3ヶ月は最難関（初期不良・オペ未確立・レビューゼロ）。**初期レビューの速度が生命線**: 早期ゲストへの丁寧対応・レビュー依頼（Airbnb＋Google）を最優先
- 週次で2棟(Airstar)との比較記録: 稼働・レビュー平均・対応事故数・本人実働時間

## 判断ゲート（運営6ヶ月後）

| 指標 | 合格ライン |
|---|---|
| 稼働率 | Airstar棟と同等以上（季節調整後） |
| レビュー平均 | 4.6以上 かつ Airstar棟と同等以上 |
| 対応事故（謝罪・補償に至った件数） | 月1件以下 |
| 本人実働時間 | 月15h以下（1棟あたり） |

**全て合格 → 4・5号棟も自社運営（構成Yをスケール）／1つでも不合格 → 差分を特定し、4・5号棟は委託を継続して再挑戦の時期を判断**
